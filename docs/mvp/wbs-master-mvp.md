# LLM 사용량 제어 시스템 - MVP 총괄 WBS

## MVP 개요

- **프로젝트명:** LLM 사용량 제어 시스템 (MVP)
- **개발 기간:** 2주 (1 M/M)
- **개발 방법론:** 단순화된 Waterfall (Mockup 기반)
- **투입 인력:** B/E(1), F/E(1)
- **총 예상 M/M:** 1.0 M/M

---

## MVP 제약 조건

1. **연동 절차 무시:** B/E와 F/E는 독립적으로 개발, 실제 API 연동 없이 Mockup 데이터 활용
2. **인력 구성:** B/E 1명 (0.5 M/M), F/E 1명 (0.5 M/M)
3. **기간:** 2주 (1 M/M) 내 완료
4. **기능 범위:** 핵심 기능 위주, 고급 기능 제외

### PHASE 2: 백엔드 개발 (Day 3-7)
**기간:** 5일
**담당:** B/E 개발자 (1명)
**소요 M/M:** 0.5

#### WBS-MVP-02: 백엔드 개발

| Task ID | 작업 내용 | 상세 내용 | 기간 | 산출물 |
|---------|----------|----------|------|--------|
| MVP-02-01 | 프로젝트 초기화 | Spring Boot 프로젝트 생성, 기본 설정 | Day 3 | 프로젝트 구조 |
| MVP-02-02 | Mock 데이터 구조 설계 | In-Memory DB 또는 JSON 파일 기반 Mock 데이터 | Day 3 | Mock 데이터 |
| MVP-02-03 | 인증 API 구현 (Mock) | 하드코딩된 로그인 로직, 토큰 발급 Mock | Day 3-4 | 인증 API |
| MVP-02-04 | 팀 관리 API 구현 (Mock) | 팀 CRUD, 팀원 관리 API, In-Memory 저장 | Day 4-5 | 팀 관리 API |
| MVP-02-05 | 모델 관리 API 구현 (Mock) | 모델 등록/조회 API, Mock 데이터 반환 | Day 5-6 | 모델 관리 API |
| MVP-02-06 | 사용량/대시보드 API 구현 (Mock) | 할당량, 대시보드 데이터 Mock 반환 | Day 6-7 | 사용량/대시보드 API |
| MVP-02-07 | API 명세서 작성 | Swagger 또는 Postman Collection | Day 7 | API 명세서 |

**참고:**
- 실제 DB 연동 없이 In-Memory DB (H2) 또는 JSON 파일 활용
- 모든 데이터는 Mockup으로, 실제 LLM API 연동 없음
- 기본 CRUD만 구현, 비즈니스 로직은 단순화

- 실제 B/E API 연동 없이 Mock 데이터 활용
- json-server 또는 로컬 JSON 파일 사용 가능
- 기본 UI 레이아웃과 필수 기능만 구현
