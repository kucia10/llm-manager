# RFP 작성
## 요건 정의
SRS(Software Requirements Specification) 기준으로 RFP(Request for Proposal)를 작성합니다.

### 목표 시스템
- **시스템명**: LLM 사용량을 제어하는 Admin
- **시스템 목적**: LLM(Large Language Model)의 사용량을 관리하고 제어하는 관리자 시스템

### 개발 정보
- **백엔드**: Spring Boot
- **프론트엔드**: React.js
- **프로그램 타입**: 웹 프로그램
- **통신 방식**: RESTful API 방식

### 주요 기능 상세
- **로그인 정보**
  - OAuth 2.0 기반 인증 시스템
  - 사용자 인증 및 권한 관리
  
- **팀 관리 기능**
  - 팀 생성 및 관리
  - 팀원 관리
  - 팀별 권한 설정
  
- **LLM 모델 등록/관리**
  - LLM 모델 정보 등록
  - 모델별 속성 관리
  - 모델 활성/비활성 관리
  
- **사용량 통계 정보**
  - 전체 사용량 집계
  - 기간별 사용량 추이
  - 사용자별/팀별 사용량 분석
  
- **팀별/개인별 사용량 제어**
  - 팀별 할당량 설정
  - 개인별 할당량 설정
  - 사용량 초과 시 알림 및 제어
  
- **사용량 통계 확인을 위한 대시보드**
  - 실시간 사용량 시각화
  - 사용량 차트 및 그래프
  - 주요 지표(KPI) 표시
  
- **비용 관리**
  - 사용량 기반 비용 산정
  - 팀별/개인별 비용 배분
  - 비용 보고서 생성

# RFP 분석 / User Story 작성
RFP를 기준으로 구현이 필요한 항목을 도출하여 User Story를 구성합니다.

### 문서 위치
- **저장 경로**: `docs/init` 폴더

### 프로젝트 기본 정보
- **프로젝트 기간**: 6개월
- **인력 지원**: 개발에 필요한 인력 충분히 지원 가능한 상태

### 산정 항목
- User Story 도출
- 각 User Story별 투입 M/M(Man-Month) 산정

# 파트별 Task 관리
User Story를 기준으로 업무 파트별 Task를 생성하고 관리합니다.

### 관리 파트
- **PM (Project Management)**: 프로젝트 총괄 및 일정 관리
- **기획**: 요건 분석 및 기획 문서 작성
- **디자인**: UI/UX 디자인 및 시안 작업
- **퍼블**: HTML/CSS 마크업 및 스타일링
- **F/E (Frontend)**: 프론트엔드 개발
- **B/E (Backend)**: 백엔드 개발

### 문서화 항목
- **총괄 WBS**: PM이 관리하는 전체 프로젝트 WBS(Work Breakdown Structure)
- **파트별 WBS**: 각 파트별 상세 WBS

### 개발 방법론
- **방식**: Waterfall 방법론
- **특징**: 순차적 개발 프로세스 적용

### 저장 위치
- **파일 경로**: `docs/task` 폴더

# MVP 구성
MVP(Minimum Viable Product) 구성 요건으로 WBS를 작성합니다.

### 개발 범위
- **기간**: 1M/M 이내에서 수행
- **인력 구성**:
  - B/E: 1명
  - F/E: 1명

### 개발 방식
- **연동 절차**: 연동 절차는 무시하고 Mockup으로 구성
- **목적**: 핵심 기능 위주의 최소 기능 제품 구현

### 저장 위치
- **파일 경로**: `docs/mvp` 폴더

# 클라이언트 환경 구성
최신 프론트엔드 아키텍처를 기반으로 현재 폴더(demo) 아래에 클라이언트 개발 환경을 구축합니다.

### 기술 스택
- **프레임워크**: React
- **빌드 도구**: Vite
- **스타일링**: Tailwind CSS

### 구성 항목
1. **프로젝트 스캐폴딩 생성**
   - React + Vite 기반 프로젝트 초기화
   - 성능 및 개발자 경험 최적화 설정

2. **Tailwind CSS 완전 통합**
   - Tailwind CSS 설정 파일 구성
   - PostCSS 구성
   - 유틸리티 클래스 전역 적용 설정

3. **모듈형 폴더 구조 구현**
   - 컴포넌트 폴더: 재사용 가능한 UI 컴포넌트
   - 페이지 폴더: 각 페이지별 컴포넌트
   - 에셋 폴더: 이미지, 스타일 등 정적 리소스
   - 공통 모듈: 유틸리티, 서비스, 컨텍스트 등

### 구성 목표
- **확장성**: 기능 추가 시 쉽게 확장 가능한 구조
- **유지보수성**: 코드의 가독성과 관리 용이성 확보

### 최종 작업
- 개발 서버 구동 및 정상 동작 확인

### 저장 위치
- **프로젝트 폴더**: `mvp-client` 폴더
